# Axiom Details & Examples

Expanded descriptions and examples for axioms that benefit from further explanation.

# 0 - Meta & orchestration

## X000

Make automatable checks instead of axioms.

If a rule can be enforced by a linter, formatter, or CI check, don’t make it an axiom. Axioms are reserved for judgment calls that require human reasoning.

## X002

Choose a harmonious software stack.

A harmonious stack means the tools, frameworks, and languages you choose should complement each other rather than fight. Harmony reduces glue code, mental context-switching, and configuration surface area.

✅ Good: Django + Postgres + Redis + Celery — Django has first-class support for Postgres features, and Celery is the de facto async task runner for Django apps.

❌ Bad: Django + MongoDB + Sidekiq — MongoDB fights Django’s ORM, and Sidekiq is a Ruby tool with no natural bridge to Python.

## X003

Max out formatting, linting, and static analysis tools, with opt-in-by-default for rules.

For every language in your stack, adopt the strictest available formatter, linter, and static analysis tool. Automated tools catch entire categories of bugs and style drift before code review even begins, freeing reviewers to focus on design and logic.

✅ A well-covered Python, Django, HTML & CSS stack:

| Tool | Role | Scope |
| --- | --- | --- |
| ruff | Linter & formatter | Python |
| ty | Static type checking | Python |
| wemake-python-styleguide | Opinionated complexity & style linter | Python |
| Cutesy | Formatter | HTML templates |
| Prettier | Formatter | CSS, JSON, Markdown |
| ESLint | Linter | JavaScript |
| SwiftLint | Linter & style enforcer | Swift |

❌ Gaps to avoid:

- Running ruff with rules disabled — you get formatting but miss real bugs.
- Skipping type checking because the codebase started untyped — add ty and type incrementally.
- Having no linter for templates or stylesheets — HTML and CSS errors slip through just as easily as Python ones.
Wire every tool into a single make lint (see X004) and require it to pass in CI (see X107).

## X004

Centralize automation commands, with modular entry points.

Use a single command runner as the entry point for all project tasks. Each command should be small and composable.

```Makefile
lint:
	ruff check .
	ty check .

test:
	pytest

ci: lint test

deploy:
	./scripts/build.sh && ./scripts/deploy.sh
```

## X005

Solve problems at the right level of abstraction.

Before writing application code, ask whether the problem is better solved at a lower layer (CDN, database, OS) or a higher one (configuration, infrastructure).

Example — Rate limiting: You could write middleware in Django, but if you’re behind Cloudflare, a WAF rule solves it with zero application code and better performance.

Example — Uniqueness: Enforce unique constraints at the database level, not just in application validation. The database is the single source of truth.

## X006

Use AI in workflows where the output is fully verifiable. Give agents the loop.

AI is valuable & safe where you have a strong verification layer already in place — the same CI pipeline, type checker, and test suite that catches human mistakes catches agent mistakes. The stronger your automated checks (X003, X107, X301), the more safely you can delegate to agents.

Critically, don’t just verify the output after the fact — give the agent access to the verification tools during its work. An agent that can run make lint, see that ty found a type error on line 42, and fix it before presenting the result is dramatically more useful than one that produces a single shot and hopes for the best. The iteration loop is the product:

1. Agent makes a change.
1. Agent runs the checks.
1. Agent reads the failure output.
1. Agent fixes the issue.
1. Repeat until clean.
Design your automation commands (X004) so agents can invoke the same make lint, make test, and make build entry points that humans use.

## X007

Use the filesystem as the agent’s memory. Use progressive discovery to avoid bloating context.

Structure knowledge in the filesystem so agents can find and read what’s relevant as they work — the same way a new developer would navigate the project. In a short top-level agent context file like CLAUDE.md, instruct agents to use this system for their own benefit.

```plain text
.claude/
  settings.json        # Agent config & permissions
  skills/              # Learned patterns, codebase-specific knowledge
    naming.md
    test-patterns.md

docs/agents/
  conventions.md       # Project-level rules agents should follow
  examples/            # Reference outputs for common tasks
    migration.md
    api-endpoint.md
```

.claude/ (or equivalent) is agent-managed working memory — skills it’s learned, scratchpad notes, codebase-specific patterns it’s picked up. Let agents write here so they improve over time. docs/agents/ is human-curated guidance that agents read but don’t typically modify. Standard project docs (README.md, CONTRIBUTING.md, inline docstrings) serve both audiences.

Progressive discovery means the agent reads the top-level README, learns there’s a docs/ directory, reads what’s relevant to the current task, and drills deeper only as needed. This mirrors how context-limited agents actually work best — small, relevant context windows rather than an entire codebase in the system prompt.

## X008

Give agents access to a full development information set via MCPs and other discoverable interfaces.

Expose the same information surfaces that a human developer uses:

- MCP servers for browser access, API docs, issue trackers, hosted version control, and other live project context.
- CLI tools for builds, tests, linting, and deploys (see X006).
- Search interfaces for codebase search, documentation search, and log search.
Well-designed MCP integrations make the agent’s environment self-documenting — it can ask what’s available and query what it needs.

## X009

Version agent skills and shared context alongside the code they operate on.

Agent skills files, project-level conventions in docs/agents/, and other shared agent context are configuration — treat them like infrastructure (X113). When agent behavior needs to change because the codebase changed, the instruction update should be in the same commit. This keeps agent behavior reproducible, reviewable, and tied to the code it operates on. The .claude/skills/ directory and docs/agents/ conventions should be checked into version control and reviewed in PRs just like any other project configuration.

## X110

Use centralized logging, but avoid noisy or low-value logs.

Log meaningful events: errors, state transitions, external calls. Don’t log every step of a routine operation.

```Python
# ❌ Noisy
logger.info(f"Checking user {user.id}")
logger.info(f"User {user.id} found")

# ✅ Meaningful

```

# 2 - Code

## X200

Name things concisely and descriptively.

A name should tell you what something is or does at a glance. Avoid both cryptic abbreviations and verbose names.

```Python
# ❌ Too terse
def proc(data):
    return data.get("val")

# ❌ Too verbose
def process_incoming_data_and_extract_primary_value(incoming_data_dictionary):
    return incoming_data_dictionary.get("value")

# ✅ Concise and descriptive
def extract_value(data):
    return data.get("value")
```

## X201

Do not include project prefixes in variable names.

The module or package already provides namespace context.

```Python
# ❌ Redundant prefix
class BringFidoUserProfile:
    bf_user_email = ...

# ✅ Let the module or project provide context
# bringfido/models.py
class UserProfile:
    email = ...
```

## X202

Use variable tense to imply type.

The grammatical form of a name should hint at its type without needing a suffix like _list or _flag.

```Python
# Booleans — use adjectives or past participles
is_active = True
has_permission = True
email_verified = False

# Collections — use plurals
users = User.objects.all()
selected_ids = [1, 2, 3]

# Single items — use singulars
user = User.objects.get(pk=1)

# Only add the type when it differs from what's implied
user_count = len(users)        # "users" implies a collection; "count" clarifies this is a number
user_map = {user.id: user for user in users}  # "map" clarifies this is a dict, not a list
```

## X203

Avoid abbreviations except for a small allowlist.

Some abbreviations of long words are common enough in the coding world to be easily understood. Beyond those, prefer descriptive names that can be understood at a glance.

These abbreviations are OK and preferred:

ad, abbr, attr, bool, config, db, desc, dict, enum, env, expr, hex (oct, etc.), info, int, kg (lbs, ft, mi, etc.), len, max, min, num, param, pk, src, str, utils

So are abbreviations that are more descriptive on their own, especially for HTML attributes:

alt, href, src

Abbreviations using common acronyms for formal or technical names are also OK:

amex, ip, js (css, html, json, xml, etc.), url, USD (CAD, etc.)

Avoid abbreviations that are uncommon, ambiguous, or save almost no effort over the full word:

Don’t use these: ctx, err, idx, obj, val, exc

In most other cases, use the full word for clarity.

When an acronym appears in a variable name, use consistent casing across the whole name, for example:

- url, id
- projectID, websiteURL
## X204

Write as little code as necessary.

Every line is a liability — it must be read, understood, tested, and maintained. Prefer built-in features, standard library functions, and well-maintained packages over custom implementations.

```Python
# ❌ Reimplementing what exists
def flatten(nested_list):
    result = []
    for sublist in nested_list:
        for item in sublist:
            result.append(item)
    return result

# ✅ Use what's available
from itertools import chain

...

flat = list(chain.from_iterable(nested_list))
```

## X208

Avoid circular dependencies between packages & modules, not just between individual files.

Packages and modules should have a clean dependency tree between them, and another clean one within them. If package A imports from package B, then B must not import from A — directly or transitively. Apply the same discipline within a package: if module X imports module Y, Y should not reach back into X.

Circular dependencies make it impossible to understand, test, or deploy a package in isolation. They also cause subtle import-time bugs in Python and linker issues in compiled languages.

```plain text
# ❌ Circular dependency between packages
orders/
    models.py      → imports from payments.models
payments/
    models.py      → imports from orders.models

# ✅ Clean dependency tree between packages
orders/
    models.py      → imports from payments.models
payments/
    models.py      → no knowledge of orders
```

```plain text
# ❌ Circular dependency within a package
orders/
    models.py      → imports from orders.utils
    utils.py       → imports from orders.models

# ✅ Clean dependency tree within a package
orders/
    models.py      → imports from orders.utils
    utils.py       → standalone helpers, no imports from orders.models
```

Don’t use inline imports. All imports should be at the top of the file. This is only sustainable when the dependency graph is clean — if moving an inline import to the top of the file causes an ImportError, that’s a sign of a circular dependency that needs to be fixed. Removing inline imports is a useful way to surface hidden cycles.

Use a tool like pydeps to visualize the dependency graph. Run it at the package level to get a high-level map, and within a package to see module-level relationships. Cycles and unexpected edges are immediately obvious in the visual output.

```Bash
# Visualize inter-package dependencies
pydeps myproject --cluster --max-bacon=2

# Visualize intra-package dependencies for a single package
pydeps myproject.orders --no-show --output orders-deps.svg
```

When you discover a cycle, break it by:

- Extracting shared logic into a lower-level module that both sides can depend on.
- Inverting the dependency with a callback, protocol, or event so the lower-level module doesn’t need to know about the higher-level one.
- Merging tightly coupled modules that genuinely belong together.

## X212

Avoid circular “self” or “this” references.

Don’t set or modify attributes on self in a helper method that’s called by another method, and then depend on that value in a peer method of the caller. This creates hidden data flow through self that makes the calling method’s behavior dependent on the order of its sub-calls — a circular reference through the instance’s own state.

Even peer methods at the same level should have a clear depth order: if method A and method B are both called by method C, A should not silently set state on self that B later reads. Data should flow through return values or explicit parameters, not through side effects on self.

> 

```Python
# ❌ Hidden data flow through self
class OrderProcessor:
    def process(self):
        self._validate()       # Silently sets self.discount
        self._apply_pricing()  # Reads self.discount — breaks if called first

    def _validate(self):
        self.discount = self._calculate_discount()

    def _apply_pricing(self):
        self.total = self.subtotal - self.discount  # Depends on _validate running first

# ✅ Explicit data flow
class OrderProcessor:
    def process(self):
        discount = self._calculate_discount()
        self.total = self._apply_pricing(discount)

    def _calculate_discount(self) -> Decimal:
        ...

    def _apply_pricing(self, discount: Decimal) -> Decimal:
        return self.subtotal - discount
```

## X213

Use composed methods and small functions to express intent clearly.

Each function should do one thing at one level of abstraction. The calling function reads like a summary; the called functions contain the details.

```Python
# ❌ One long function
def process(self):
    # 40 lines validating, charging, emailing, updating inventory...
    pass

# ✅ Composed methods
def process(self):
    self._validate_order()
    self._charge_payment()
    self._update_inventory()
    self._send_confirmation()
```

## X215

Use fat models for structural and domain logic.

Business rules that relate to what an entity is or can do belong on the model, not in views or serializers. This keeps logic testable and reusable.

```Python
# ❌ Logic in the view
def cancel_order_view(request, order_id):
    order = Order.objects.get(pk=order_id)
    if order.status == "shipped":
        return HttpResponseBadRequest("Cannot cancel")
    order.status = "cancelled"
    order.cancelled = timezone.now()
    order.save()

# ✅ Logic on the model
class Order(models.Model):
    def cancel(self):
        if self.status == "shipped":
            raise OrderError("Cannot cancel a shipped order")
        self.status = "cancelled"
        self.cancelled = timezone.now()
        self.save()
```

## X216

Use adapters for integration logic.

Wrap third-party APIs and services in thin adapter modules. The rest of your code should avoid calling an external API directly. This isolates change, simplifies testing, and makes swapping providers straightforward.

```Python
# adapters/email.py
def send_email(to, subject, body):
    """Thin wrapper around the email provider."""
    return ses_client.send_email(
        Destination={"ToAddresses": [to]},
        Message={"Subject": {"Data": subject}, "Body": {"Text": {"Data": body}}},
    )

# orders/models.py — depends on the adapter, not on SES directly
from adapters.email import send_email

class Order(models.Model):
    def send_confirmation(self):
        send_email(self.user.email, "Order confirmed", self.confirmation_body)
```

Adapters can also be inheritable classes that models mix in, keeping integration logic colocated with the model but separated into its own layer:

```Python
# adapters/braze.py
class BrazeOrder:
    """Adapter mixin that syncs order status to Braze."""

    def _sync_status_to_braze(self):
        braze_client.track_user(
            self.user.external_id,
            attributes={"order_status": self.status},
        )

    def save(self, *args, **kwargs):
        super().save(*args, **kwargs)
        self._sync_status_to_braze()


# orders/models.py
from adapters.braze import BrazeOrder

class Order(BrazeOrder, models.Model):
    status = models.CharField(max_length=20)
    # ...

    def cancel(self):
        self.status = "cancelled"
        self.save()  # Braze sync happens automatically via the adapter
```

The model inherits from BrazeOrder, which hooks into save() to sync the order status to Braze. The model itself has no knowledge of the Braze API — that concern lives entirely in the adapter.

Adapters are also the right place for content formatting, such as building JSON representations for external consumers:

```Python
# adapters/api.py
from datetime import datetime
from decimal import Decimal

from django.db import models


class APIOrder:
    """Adapter mixin that provides a JSON-serializable representation for API responses."""

    pk: int
    status: str
    total: Decimal
    created: datetime
    items: models.Manager

    @property
    def json_data(self) -> dict:
        return {
            "id": self.pk,
            "status": self.status,
            "total": str(self.total),
            "createdAt": self.created.isoformat(),
            "items": [
                {"sku": item.sku, "quantity": item.quantity}
                for item in self.items.all()
            ],
        }


# orders/models.py
from adapters.api import APIOrder
from adapters.braze import BrazeOrder

class Order(APIOrder, BrazeOrder, models.Model):
    status = models.CharField(max_length=20)
    # ...


# orders/views.py
def order_detail(request, order_id):
    order = Order.objects.get(pk=order_id)
    return JsonResponse(order.json_data)
```

This keeps serialization logic out of views and models. If the API contract changes or a second format is needed (e.g., for a webhook payload), the adapter is the only thing that changes.

## X217

Prefer declarative over imperative.

Declarative code describes what the result should be. Imperative code describes how to get there step by step. Declarative code is typically easier to read, less error-prone, and more composable.

```Python
# ❌ Imperative
active_emails = []
for user in users:
    if user.is_active:
        active_emails.append(user.email)

# ✅ Declarative
active_emails = [user.email for user in users if user.is_active]
```

```HTML
<!-- ❌ Imperative (vanilla JS) -->
<button onclick="document.getElementById('menu').style.display='block'">Open</button>

<!-- ✅ Declarative (Alpine.js) -->
<div x-data="{ open: false }">
  <button @click="open = true">Open</button>
  <nav x-show="open">…</nav>
</div>
```

Don’t repeat lookups in the same scope.

Capture the result of an expression once and reuse it. Don’t incrementally build up objects on self — construct them fully, then assign.

```Python
        self.label = f"{self.product.category.name} - {self.product.name}"

# ✅ Captured once
def update_label(self):
    category_name = self.product.category.name

```

```Python

# ✅ Build first, then assign
```

## X221

Prefer rich data types and explicit structures to dicts or tuples with implied structure.

Named structures make code self-documenting and catch errors at definition time rather than at access time.

## X223

Minimize intermediate state classes.

Avoid creating dedicated classes whose only purpose is to shuttle data between two steps of a pipeline. Classes like EvaluationResult, ModerationDecision, or ValidationOutcome often hold a handful of fields that belong on the primary model or can be returned as a simple tuple or dataclass local to the function.

When every processing step produces its own bespoke result type, the codebase fills with thin wrappers that obscure the actual data flow. Readers have to chase through layers of indirection to find out what’s really being passed around.

```Python
# ❌ Intermediate state classes that duplicate model fields
class EvaluationResult:
    def __init__(self, score, passed, evaluated_at):
        self.score = score
        self.passed = passed
        self.evaluated_at = evaluated_at

class ModerationDecision:
    def __init__(self, approved, reason, moderated_at):
        self.approved = approved
        self.reason = reason
        self.moderated_at = moderated_at

def process_submission(submission):
    result = evaluate(submission)           # returns EvaluationResult
    decision = moderate(submission, result)  # returns ModerationDecision
    submission.score = result.score
    submission.passed = result.passed
    submission.approved = decision.approved
    submission.moderation_reason = decision.reason
    submission.save()
```

```Python
# ✅ Let the model own the state; methods mutate directly
class Submission(models.Model):
    score = models.FloatField(null=True)
    passed = models.BooleanField(null=True)
    evaluated_at = models.DateTimeField(null=True)
    approved = models.BooleanField(null=True)
    moderation_reason = models.TextField(blank=True)
    moderated_at = models.DateTimeField(null=True)

    def evaluate(self):
        self.score = self._calculate_score()
        self.passed = self.score >= self.PASSING_THRESHOLD
        self.evaluated_at = timezone.now()

    def moderate(self):
        self.approved = self._check_content_policy()
        self.moderation_reason = "" if self.approved else self._explain_rejection()
        self.moderated_at = timezone.now()
```

When a separate class is warranted:

- The result crosses a system boundary (e.g., returned from an external API adapter).
- The intermediate structure has its own identity or lifecycle and is queried independently.
- Multiple unrelated consumers need the same computed shape.
If the class is only created to be immediately unpacked into the model that triggered the computation, skip the middleman.

## X224

Use highly specific type signatures. Don’t allow None where a value is always expected.

Every Optional field or nullable parameter that can never actually be None at runtime is a lie in the type signature. It forces every caller to handle a case that doesn’t exist, adding pointless if x is not None guards, cluttering code, and hiding the real invariants of the system.

Narrow types to exactly what’s possible. If a value is always present after construction, don’t mark it optional. If a function always returns a result, don’t return None. If a field is set during __init__ and never cleared, type it as the concrete type.

```Python
# ❌ Overly permissive — None is never valid here
class User:
    email: str | None = None  # Every user must have an email

    def send_welcome(self):
        if self.email is None:  # This can never happen
            raise ValueError("No email")
        send(self.email)

# ✅ Type reflects reality
class User:
    email: str  # Required at construction — no None checks needed

    def send_welcome(self):
        send(self.email)
```

```Python
# ❌ Return type is broader than reality
def get_config() -> dict | None:
    return {"timeout": 30}  # Always returns a dict

config = get_config()
if config is not None:  # Dead branch
    timeout = config["timeout"]

# ✅ Return type matches behavior
def get_config() -> dict:
    return {"timeout": 30}

timeout = get_config()["timeout"]
```

This applies beyond None. Prefer str over str | int if the value is always a string. Prefer a concrete dataclass over dict if the shape is known. Prefer an enum over str if the values are constrained. The narrower the type, the less defensive code you need and the more the type checker can verify for you.

## X225

Phrase method names imperatively. Distinguish subject methods from object methods.

Every method name should read as a command. How you phrase it depends on the role self plays in the action.

Subject methods — self is the actor, performing an action on something passed in. The primary arguments or return values are direct objects of the verb; remaining arguments are prepositional.

```Python
# self acts ON the argument
processor.validate(order)
serializer.serialize(queryset)
emailer.send(message, to=recipient)
converter.transform(data, into=format)
factory.make_widget(color="red")
```

Object methods — self is the thing being acted on. The caller performs the action on the object. All arguments are prepositional, because there is no direct object — self is the object.

```Python
# caller acts ON self
order.save()
order.cancel(reason="duplicate")
user.authenticate(token=token)
report.export(to="csv")
```

Use verb prefixes to clarify intent when the bare verb would be ambiguous — especially for methods whose primary purpose is to produce or retrieve a value rather than cause a side effect:

```Python
# ❌ Ambiguous — is this a noun or a command? Does it have side effects?
order.total()
user.name()

# ✅ Prefix clarifies: these retrieve/compute a value
order.get_total()
user.get_display_name()

# ✅ Prefix clarifies: these produce a new object
factory.make_order(for_user=user)
builder.build_query(from_params=params)

# ✅ Prefix clarifies: these fetch from an external source
service.fetch_rates(for_currency="USD")
client.fetch_user(user_id)
```

Bare verbs are best when the action is obvious and side-effectful:

```Python
# ✅ No prefix needed — the verb is clear and imperative
order.cancel()
queue.flush()
cache.invalidate(key)
user.deactivate()
```

The key test: read the call site aloud. processor.validate(order) reads as a command — "processor, validate this order." order.save() reads as a command — "order, save yourself." If the method name doesn’t read as a clear imperative, rephrase it.

# 3 - Tests

## X300

Refactor toward the core principles before adding tests, so you test the right shapes.

If the code is poorly structured, writing tests first locks in the wrong design. Refactor to clean boundaries and clear responsibilities, then write tests against those shapes.

Example: You inherit a view that queries the database, applies business rules, formats a response, and sends an email — all inline. Writing tests for this view cements the coupling. Instead, extract the business logic into a model method (X215), the email into an adapter (X216), and the query into a manager. Then write focused tests for each piece.

## X301

Maintain separate test suites for smoke, unit, integration, and contract tests.

Each suite has a different purpose and speed:

- Smoke tests: Quick checks that core paths work. Run on every deploy.
- Unit tests: Fast, isolated tests for individual functions and methods.
- Integration tests: Verify that components work together (database, API calls).
- Contract tests: Test the public interface of a package, ensuring backwards compatibility.
## X302

Test state should be fully independent between tests. Use database isolation.

Each test must start from a clean, predictable state. Tests that leak database rows, cache entries, or file-system artifacts into other tests create flaky, order-dependent suites that are painful to debug. Use a test runner or base class that rolls back or flushes the database between tests.

In Django, TransactionTestCase resets the database by truncating tables after each test, giving every test a truly empty database. The default TestCase wraps each test in a transaction and rolls it back, which is faster but can hide bugs involving real commits, signals, or on_commit hooks.

```Python
# ❌ Shared state between tests — order-dependent and fragile
class OrderTests(TestCase):
    @classmethod
    def setUpClass(cls):
        super().setUpClass()
        cls.user = User.objects.create(username="shared_user")  # Persists across all tests

    def test_place_order(self):
        order = Order.objects.create(user=self.user, status="pending")
        self.assertEqual(Order.objects.count(), 1)

    def test_no_orders_exist(self):
        # 💥 Fails if test_place_order runs first — leftover row from the other test
        self.assertEqual(Order.objects.count(), 0)
```

```Python
# ✅ Each test gets a clean database — order-independent
from django.test import TransactionTestCase

    def setUp(self):

        self.assertEqual(Order.objects.count(), 1)

    def test_no_orders_exist(self):

```

# D - Database

## D000

Use a mapper or ORM for application-facing data access.

Application code should interact with the database through an ORM, not raw SQL. This keeps queries composable, consistent with the model layer, and portable.

```Python
# ❌ Raw SQL in views

# ✅ ORM

```

## D002

Prefer wide rows over one-to-one satellite tables.

If a table has a strict one-to-one relationship with another table, the second table almost certainly shouldn’t exist. Merge its columns into the first table. Satellite tables add joins, migration complexity, and cognitive overhead for no structural benefit.

The most common symptom is a table whose only unique key is a foreign key to the parent — that’s not a relationship, it’s an awkward extension. Fold it in.

```Python
# ❌ Unnecessary satellite table
class Reservation(models.Model):
    guest = models.ForeignKey(User, on_delete=models.CASCADE)
    check_in = models.DateField()
    check_out = models.DateField()
class ReservationConfirmation(models.Model):
    reservation = models.OneToOneField(Reservation, on_delete=models.CASCADE)
    confirmed_at = models.DateTimeField()
    confirmation_number = models.CharField(max_length=20)
    confirmed_by = models.ForeignKey(User, on_delete=models.SET_NULL, null=True)
```

```Python
# ✅ Wide row — all data lives on the primary model
class Reservation(models.Model):
    guest = models.ForeignKey(User, on_delete=models.CASCADE)
    check_in = models.DateField()
    check_out = models.DateField()
    confirmed_at = models.DateTimeField(null=True, blank=True)
    confirmation_number = models.CharField(max_length=20, blank=True)
    confirmed_by = models.ForeignKey(
        User, on_delete=models.SET_NULL, null=True, blank=True, related_name="confirmed_reservations",
    )
```

When to keep a separate table:

- One-to-many or many-to-many relationships — those aren’t satellite tables, they’re real relations.
- Polymorphic or multi-type records — e.g., a Payment that can belong to different parent types via a shared interface.
- Genuinely independent lifecycle — the child entity is created, queried, and deleted independently of the parent.
If the only reason for a second table is "it felt like a separate concept," that’s not enough. Concepts are cheap; tables and joins are not.

# P - Python

## P000

Ask for forgiveness, not for permission (EAFP).

In Python, it’s typically better to attempt an operation and handle the exception than to check preconditions first. This is idiomatic, often faster when the happy path is common, and avoids race conditions.

```Python
# ❌ LBYL (Look Before You Leap)
if "key" in data:
    value = data["key"]
else:
    value = default

# ✅ EAFP (Easier to Ask Forgiveness than Permission)
try:
    value = data["key"]
except KeyError:
    value = default

# ✅ Even simpler — use the built-in that embodies this
value = data.get("key", default)
```

## P001

Let unexpected exceptions bubble to a centralized handler.

Don’t wrap code in broad try/except blocks that swallow errors. Let unexpected exceptions propagate to Sentry (or equivalent).

```Python
# ❌ Swallowing errors
try:
    process_payment(order)
except Exception:
    pass

# ✅ Handle expected errors; let the rest bubble
try:
    process_payment(order)
except PaymentDeclinedError:
    notify_user(order, "Payment was declined")
```

## P003

Minimize the size and complexity of try blocks. Prefer a single line in the try.

A try block should wrap the smallest possible unit of code — ideally one line. When a try block contains multiple statements, it becomes unclear which line is expected to raise, and unrelated exceptions can be silently caught by the wrong handler.

```Python
# ❌ Too much in the try block
try:
    user = fetch_user(user_id)
    orders = fetch_orders(user)
    total = sum(order.amount for order in orders)
    send_summary(user.email, total)
except ConnectionError:
    log.warning("Connection failed")
# Which call raised? fetch_user? fetch_orders? send_summary?

# ✅ Wrap only the line you expect to fail
user = fetch_user(user_id)
orders = fetch_orders(user)
total = sum(order.amount for order in orders)

try:
    send_summary(user.email, total)
except ConnectionError:
    log.warning(f"Failed to send summary to {user.email}")
```

```Python
# ❌ Multiple potential failure points hidden in one try
try:
    config = json.loads(raw_config)
    db = connect(config["db_url"])
    db.execute(migration_sql)
except Exception:
    log.error("Setup failed")

# ✅ Each risky operation isolated with its own handling
try:
    config = json.loads(raw_config)
except json.JSONDecodeError:
    raise ConfigError("Invalid JSON in config file")

db = connect(config["db_url"])
db.execute(migration_sql)
```

Small try blocks make the intent obvious: this specific line can fail, and here’s how we handle it. Everything else stays outside the try so that unexpected exceptions bubble up (P001) instead of being accidentally caught.

## S004

Use the factory and context pattern for composition.

Instead of passing individual dependencies, use a factory (a caseless enum that knows how to create concrete types) and a context (shared app state). Each type declares exactly what it needs from both via protocol composition, so dependencies stay explicit and narrowly scoped even as the graph grows. See the linked article for the full pattern.

The factory is a caseless enum that conforms to every factory protocol in the app via extensions. Because it has no cases, it can never be instantiated — it exists purely as a namespace for type resolution.

```Swift
public enum DefaultFactory {}

extension DefaultFactory: EntityFactory { }
extension DefaultFactory: UserFactory { }
extension DefaultFactory: ReviewFactory { }
extension DefaultFactory: RoomFactory { }
extension DefaultFactory: RateQuoteFactory { }
// … one extension per factory protocol
```

Each type declares its requirements using Context and Factory typealiases built from protocol composition (&). This means a type only sees the slice of the context and factory it actually needs.

```Swift
public protocol PasskeyRegistrationVerificationForm: LoginForm {
    typealias Context = HTTPDataSourceManager & UserAccount.Context
    typealias Factory = DirectInputFactory
        & RequiredValidatorFactory
        & EmailValidatorFactory
        & UserAccountFactory
        & UserAccount.Factory
}
```

At the call site, the context and factory are passed together. The caller doesn’t need to know which specific dependencies the type requires — the type’s own Context and Factory typealiases handle that.

```Swift
/* Read current user from User Defaults */
AppContext.main.user = DefaultFactory.UserAccount.init(
    from: AppContext.main,
    factory: DefaultFactory.self
)

if AppContext.main.user == nil {
    UniversalSearch().removeSearchableReservations()
}
```

AppContext.main is the context — it holds shared app state like the current user. DefaultFactory.self is passed as the factory — a caseless enum that conforms to all factory protocols, so any type can resolve its concrete dependencies through it. The UserAccount initializer receives both, reads what it needs from the context, and uses the factory to create any further dependencies without the caller knowing the details.
